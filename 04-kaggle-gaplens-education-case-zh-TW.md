## 1. 案例概觀：GapLens 是什麼？

### 1.1 比賽背景

- 平台：Kaggle  
- 活動名稱：Vibe Code with Gemini 3 Pro  
- 主題類別：
  - Science（科學研究）  
  - Education（教育）  
  - Accessibility（無障礙輔助）  
  - Health（健康醫療）  
  - Business（商業應用）  
  - Tech Breakthroughs（技術突破）

- 核心規則：
  1. 必須在 Google AI Studio「Build」環境中完成 App。  
  2. 必須使用 Gemini 3 Pro，善用進階推理與原生多模態能力。  
  3. 必須提交三項成果：Kaggle Writeup、公開 App 連結、最長 2 分鐘的 Demo 影片。

### 1.2 GapLens 的定位與邏輯

GapLens – Learning Loss Scanner 的定位是：

> 給老師與補救教學者使用的「學習落差掃描器」。  
> 老師上傳學生的考卷或筆記，系統標出學生真正卡住的地方，  
> 並一鍵產出差異化作業（personalized homework pack）。

刻意的設計選擇：

- 不做完整 LMS（學習管理系統）。  
- 不做大型題庫與排程系統。  
- 單點聚焦在老師最難判斷的一步：看到錯誤答案後，判斷學生到底「哪裡沒懂」。

為了讓這一步對 AI 比較「可計算」，GapLens 將錯誤分成三種類型：

1. 概念錯誤（Conceptual error）：心中模型錯了，根本沒有建立正確概念。  
2. 程序錯誤（Procedural error）：知道概念，但步驟順序、運算流程、轉換程序用錯。  
3. 粗心錯誤（Careless error）：寫錯號碼、抄錯數字、漏寫單位等。

目標是提供老師：

- 這位學生在這份考卷裡三種錯誤類型的大致比例。  
- 每一題背後對應的概念缺口與短評。  
- 一份可以直接列印或貼到 LMS 的差異化作業。

## 2. R0 視角：AI Studio + Gemini 在教育場景的 5 類常見問題

在用 Gemini 3 Pro + AI Studio 迭代 GapLens 的過程中，會遇到許多看似零碎、其實是同一類型的問題。  
從 R0 / Orchestrator 的角度，可以整理成 5 類「原則級」問題：

1. 語言錯置：中英文混雜  
2. 內部內容外洩：開發備註、JSON、系統訊息跑到畫面上  
3. 內容過長與欄位混用：一個欄位塞太多資訊  
4. 功能範圍模糊：MVP 只做一位學生，UI 卻長得像完整 LMS  
5. 範例資料與真實資料混用：Sample Data 汙染輸出

### 2.1 問題類型一：語言錯置（中英文混雜）

#### 2.1.1 問題背景

- 老師上傳的考卷多半是中文。  
- 模型在解析時，有時會部分翻成英文、部分保留中文。  
- 有些欄位是英文 UI 文案，有些直接把中文題目原樣丟上去。

#### 2.1.2 風險

- 對學生與家長來說，混合語言會降低可讀性與信任感。  
- 對國際評審或海外合作對象來說，看起來像「沒有收斂的 Prototype」。

#### 2.1.3 Prompt 設計原則

在 System Prompt 直接鎖定語言規則，例如：

- 所有使用者可見內容必須為英文。  
- 即使輸入是中文考卷，也要先理解，再用英文重新敘述題目、錯誤與說明。  
- 不要在同一句話中混用多種語言。  

可以用清楚的句子描述，例如：

- 「All user-facing content in this app must be in English only.」  
- 「When the input exam or notes are in Chinese, read and interpret them, then restate all questions, errors and explanations in English.」

---

### 2.2 問題類型二：內部內容外洩（開發備註、JSON、系統訊息跑到畫面上）

#### 2.2.1 問題背景

在 AI Studio 的一個 Prompt 裡，通常同時塞入：

- JSON schema  
- 開發註解  
- 系統行為說明  
- 使用者實際輸入的考卷或筆記

如果沒有額外限制，模型會把這些「內部說明」當成可用素材，  
直接寫進學生或老師看到的欄位裡。

#### 2.2.2 風險

- 畫面上可能出現「This is an excellent JSON schema…」之類的句子。  
- UI 看起來像 Debug 工具，而不是老師可以帶回教室用的系統。  
- 對未來導入正式系統，也會暴露太多內部設計細節。

#### 2.2.3 Prompt 設計原則

在 System Prompt 中，把「什麼不能出現在 UI」講清楚，例如：

- 不要在任何使用者可見欄位提到「prompt、JSON schema、developer notes、system messages」。  
- 題目、解析、摘要、作業內容中，都禁止出現 JSON 或程式碼格式。  

同時明確定義欄位角色，例如：

- `"question"` 欄位只放學生要看到的題目文字。  
- `"explanation"` 欄位只放給學生看的解析，不包含開發或系統說明。  
- `"teacher_summary"` 欄位只放給老師看的短摘要，不包含題目全文。

核心概念：用文字把「內部空間」與「使用者空間」切開，  
讓模型知道哪些內容是「只能讀、不能說」。

---

### 2.3 問題類型三：內容過長與欄位混用（一個欄位塞太多資訊）

#### 2.3.1 問題背景

模型常常想「一次幫你想到所有需求」，於是：

- 同一個欄位裡塞滿題目全文、解題步驟、教學建議、延伸練習等等。  
- 在手機或窄螢幕上，變成一塊卡片就捲不完。

#### 2.3.2 風險

- 老師為了看一題解析要滑很久，無法快速抓重點。  
- 重要訊息被淹沒，看不出「這個學生最需要補什麼」。  
- 未來接前端或 API 時，還要再把長文字拆回結構化欄位。

#### 2.3.3 Prompt 設計原則

在 Prompt 裡限制長度與內容種類，例如：

- 每一題的學生解析限制在 3 句以內：一句說錯在哪、一句講正確概念、一句給小提示。  
- 解析欄位不要重複題目全文。  
- 不要在學生解析裡加老師才看得懂的教學建議。  

另外用獨立欄位承載教師摘要：

- `teacher_summary` 限制為 3 個重點條列：主要概念落差、錯誤分布、後續教學建議。  
- 不重複整份考卷內容。

重點：每個欄位只負責一種「閱讀任務」，把「對學生說」與「對老師說」完全分開。

---

### 2.4 問題類型四：功能範圍模糊（MVP 只做一位學生，UI 卻長得像完整 LMS）

#### 2.4.1 問題背景

在構思時，很容易同時想到：

- 班級儀表板  
- 多次測驗的趨勢圖  
- 自動依學習落差分組  
- 學生登入與權限管理

如果沒有收斂，AI Studio 第一版 Prototype 就會什麼都沾到一點，但每個都很淺。

#### 2.4.2 風險

- Kaggle 評審在 2 分鐘 Demo 裡抓不到主軸。  
- 老師實際使用時，也看不出這工具「今天就能幫我做什麼」。  
- 技術上每一塊都做不深，很難說服任何一方繼續投入。

#### 2.4.3 Prompt 設計原則

把 MVP 範圍直接寫死在 System Prompt，例如：

- 這個版本只分析「一位學生」的一份考卷或一份筆記。  
- 不要設計或描述完整 LMS、班級儀表板、學生登入與長期追蹤功能。  
- 所有 UI 與說明都要專注在「單一學生的這一次學習落差」。

這樣可以逼迫自己與模型專注在「單一學生」體驗，  
把班級與長期追蹤留到下一個模組或下一輪實驗再做。

---

### 2.5 問題類型五：範例資料與真實資料混用（Sample Data 汙染輸出）

#### 2.5.1 問題背景

為了讓 Demo 好看，Prompt 裡通常會放一些「示範用考題」與「假學生」。  
如果沒有清楚標示，模型可能：

- 把範例題目視為真實輸入的一部分。  
- 在輸出中混入範例名字或學校。

#### 2.5.2 風險

- Demo 畫面出現並未在實際上傳考卷中出現的題目或學校名稱。  
- 對未來導入真實學校系統，會造成資料來源與隱私邊界模糊。

#### 2.5.3 Prompt 設計原則

在 Prompt 裡明確標註「哪些欄位才是真實輸入」與「哪些只是說明」，例如：

- 僅有 `uploaded_exam_text` 與 `uploaded_notes` 代表真實學生內容。  
- Prompt 當中的範例題目、假名、學校名稱全部視為教學範例，禁止出現在輸出。  
- 不主動生成真實學校名稱、班級名稱或學號；若需要稱呼學生，使用「the student」或「this learner」之類的通用稱呼。

核心概念：真實學生資料只有一條管道進來，其餘皆為 Fake Data，  
模型不能在輸出中引用這些 Fake Data 的具體標籤。

## 3. GapLens 實際流程：從單一學生到差異化作業

在套用上述五大原則後，GapLens 被收斂成一條清楚、可 Demo 的流程。

### 3.1 輸入階段（Input）

- 使用者角色：老師、補救教學者、課後輔導老師。  
- 操作步驟：
  1. 在 AI Studio App 中選擇「Exam Mode」。  
  2. 上傳一位學生的一份考卷（照片、掃描檔或純文字）。  
  3. 視需要輸入簡短背景（年級、科目、單元）。

### 3.2 分析階段（Analysis）

Gemini 3 Pro 的任務：

1. 逐題讀取題目與學生作答內容。  
2. 判斷正確與否。  
3. 對錯誤題目標記錯誤類型：Conceptual、Procedural、Careless、Misread。  
4. 推論可能缺乏的概念、技能或策略。  
5. 統整成一份「學生學習落差摘要」。

輸出包含：

- 學生概要（student_profile）：估計分數、主要強項、最需要修補的主概念。  
- 錯誤拆解清單（error_breakdown）：逐題標出錯誤類型、概念標籤、錯誤步驟、正確步驟與簡短說明。  
- 作業包（homework_pack）：若干題補強練習題。

### 3.3 差異化作業生成（Personal Homework Pack）

在掌握錯誤分布後，系統會：

- 針對最關鍵的 2–3 個概念，產生 5–6 題練習題。  
- 每題包含題目文字、答案與解析。  
- 把「學生興趣」融入題目情境，例如：籃球、Minecraft、K-Pop 等。

設計原則：

- 不試圖重教整個章節，只補關鍵缺口。  
- 題數控制在老師與學生一節課或一晚可以消化的範圍。  
- 解析避免太長，維持可讀性。

### 3.4 教師摘要與後續建議（Teacher Summary）

額外產出 teacher_summary，給老師快速判讀：

- 三種類型錯誤的大致比例。  
- 可在課堂上重新講解或強調的重點。  
- 若未來接班級層級資料，可進一步建議小組安排或複習順序。

在 AI Studio Prototype 階段，UI 設計重點：

- 單一學生路徑要走得完：上傳 → 分析 → 看結果 → 拿到作業。  
- 老師端畫面盡量在 1–2 個螢幕內看懂重點。

## 4. GapLens 在 R0 / 工作流中的位置

從整個系列來看，GapLens 的位置與關聯如下：

1. 與第 1 篇（R0 / 方法論）的關係  
   - GapLens 被視為一個獨立模組，例如 `M04_Education_GapLens`。  
   - 這個模組掛在共用 Core Kernel 上（例如使用者、帳號、資產等）。  
   - Schema、欄位、模組邊界與對 Core 的存取權限，由 R0 在 SSOT 中維護。

2. 與第 2 篇（溝通地圖）的關係  
   - 主要利害關係人是老師、補教老師、教育研究者，大致落在 L3–L5。  
   - 溝通時重點放在教學工作流程，而不是模型內部架構。

3. 與第 3 篇（MVP → 上線工作流）的關係  
   - 題目：Education / Learning Loss。  
   - MVP 範圍：單一學生的 Exam Mode。  
   - Prototype：Google AI Studio + Gemini 3 Pro。  
   - Demo：最長 2 分鐘影片，說清楚故事線與實際操作。  
   - 提交：Kaggle Writeup + App 連結 + 影片連結。

## 6. 後續延伸方向與相關篇章

如果未來要把 GapLens 從黑客松作品升級為正式產品，可能的延伸包括：

- 與真實成績系統與題庫整合，支援多次測驗與課程結構。  
- 加上班級層級視角與時間序列分析，協助學校做系統性決策。  
- 將差異化作業與既有 LMS / 作業系統整合，讓老師不用多開一個介面。

不論怎麼延伸，仍然會依賴本系列的核心原則：

- 用 R0 / SSOT / Core + Modules 控制系統複雜度。  
- 用 AI 調度串起多工具，而不是把所有責任交給單一模型。  
- 在每次實驗結束前，優先回收可重用的架構與方法，再決定是否繼續投資該產品。
