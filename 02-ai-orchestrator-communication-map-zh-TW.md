## 0. 系列地圖：本篇在整體中的位置

在系列文第 1 篇，L1–L10 是用來描述「工作本身的抽象層級」。

本篇則把 L1–L10 轉成另一個軸線：

- 用來描述「**人／團隊實際使用 AI 的成熟度**」；  
- 幫助 AI Orchestrator / AI PM 在溝通前，就先估算對方大概在哪一級；  
- 再決定要講多深、多技術，與這一場互動現實中能達到的目標。

可以把它想像成一張：

> **「跟不同等級的 AI 使用者開會時，要講到哪裡、交付什麼」的地圖。**

## 1. L1–L10：AI 使用與調度成熟度刻度

這個刻度只看「實際工作裡怎麼用 AI」，不看職稱與年資。

| Level | 標籤                     | 簡要說明                                                                 |
| ----- | ------------------------ | ------------------------------------------------------------------------ |
| L1    | 工具消費者              | 偶爾用 AI 查資料、翻譯、改寫，沒有固定流程。                            |
| L2    | 任務型使用者            | 穩定在幾種固定任務（寫信、摘要等）使用 AI，但一次只處理一件事。         |
| L3    | 多步驟玩家              | 會拆步驟、多輪對話，但流程全靠記憶與臨場反應。                          |
| L4    | 個人 SOP / Prompt 設計者 | 有自己的 Prompt/SOP 庫，自用穩定，但尚未團隊化。                        |
| L5    | 半自動流程設計者        | 會用 n8n / Make / Script 把 AI 串到其他工具，形成半自動流程。           |
| L6    | AI 應用 PM              | 會寫 AI 功能 PRD、設計流程與指標，能與工程／資料科學合作落地。         |
| L7    | AI Orchestrator PM      | 設計多 Agent、多工具、多系統的調度與分工。                              |
| L8    | AI 產品架構 PM          | 負責跨產品共用的 AI 能力平台與服務介面。                                |
| L9    | 組織級 AI Enablement    | 規劃全公司 AI 能力路線、治理規則與培訓／預算。                          |
| L10   | AI-First 轉型領航者     | 從公司策略高度，以 AI 作為核心要素進行轉型。                            |

實務上，在會議前只需要粗估一個「區間」，例如：

- L1–L2、L3–L4、L5–L6、L7–L8 或 L9–L10。  
- 目的是決定：  
  - 要不要談模型細節？  
  - 用什麼形式呈現（簡報 / Demo / PRD / 架構圖…）？  
  - 這次互動的「現實目標」是什麼。

## 2. 通用四步驟溝通框架

在進入任何一場與 AI 相關的會議前，可以先跑一次同樣的四步驟。

### 步驟一：粗估對方大概落在哪個 Level 區間

常用的三個試探問題：

1. 「你現在平常會用 AI 做哪些事情？」  
2. 「有沒有固定會重複使用的 Prompt、模板或 SOP？」  
3. 「有沒有用過 n8n、Make 或 Script，把 AI 跟其他工具串在一起？」

根據回答，大致可以判斷對方是：

- 只把 AI 當搜尋／翻譯（L1–L2）；  
- 有在設計多步驟流程（L3–L4）；  
- 已經在做自動化或功能設計（L5–L6 以上）。

### 步驟二：決定這次溝通的「主要目標」

不同 Level，目標不同：

- 面對 L1–L2：  
  - 目標是拿到**具體痛點**與**理想畫面**，而不是教模型原理。  
- 面對 L3–L4：  
  - 目標是把腦中的 SOP 抽出來，寫成可重複、可分享的流程。  
- 面對 L5–L6：  
  - 目標是得到清楚的「資料 / API 契約」與「成功指標」。  
- 面對 L7–L8：  
  - 目標是決定這個需求在「AI 能力平台」裡要放哪一層。  
- 面對 L9–L10：  
  - 目標是對齊策略、治理與投資優先順序，而不是實作細節。

### 步驟三：選擇合適的溝通載體

根據 Level，選擇不同的輸出物：

- L1–L2 → 截圖、簡單流程圖、顯示 Before/After 的 Demo。  
- L3–L4 → 步驟清單、Prompt 模板、Checklist。  
- L5–L6 → PRD、序列圖、欄位定義表、KPI 列表。  
- L7–L8 → 能力地圖、平台架構圖、Roadmap。  
- L9–L10 → 一頁紙摘要、商業案例與投資組合總覽。

### 步驟四：分成「需求會議」與「執行／落地會議」

幾乎每個專案都建議拆成兩種會議：

1. **需求／探索向（Discovery）**
   - 現在怎麼做？  
   - 哪裡最痛？  
   - 3–6 個月後覺得「成功」長什麼樣？  
   - 哪些步驟願意保留人工？  
   - 哪種 AI 錯誤可以接受，哪種絕對不能發生？

2. **執行／落地向（Delivery）**
   - 誰負責長期維護 SOP／Prompt／設定？  
   - 誰負責看 Log 與數據？  
   - 出問題時如何回退？  
   - 我們多久檢視一次（例如兩週、每月 Review）？

## 3. 各 Level 實戰摘要（壓縮版 Playbook）

以下是壓縮版的實戰指引，方便快速對照。

### L1–L2：工具消費者／任務型使用者

- **目標**：  
  - 拿到 1–2 個具體痛點與成功畫面。  
- **建議作法**：  
  - 用實際 Demo 展示「貼上內容 → 按鈕 → 結果」。  
  - 不講模型與架構，只講「你原本怎麼做，改用這個會省多少時間」。
- **產出物**：  
  - Before/After 範例、一張簡單流程圖。  

### L3–L4：多步驟玩家／個人 SOP 設計者

- **目標**：  
  - 把他腦中的多步驟流程，整理成可寫下來的 SOP。  
- **建議作法**：  
  - 請他描述「完成一件事時通常分成幾步」；  
  - 一步一步拆成文字與簡單流程圖。  
- **產出物**：  
  - 步驟清單、Prompt 模板、可重複使用的流程節點。

### L5–L6：半自動流程設計者／AI 應用 PM

- **目標**：  
  - 將需求固化成「資料結構 + API 定義 + 成功指標」。  
- **建議作法**：  
  - 一起畫出資料流與事件流；  
  - 決定哪裡需要人工複核，哪裡可以完全自動。  
- **產出物**：  
  - PRD、欄位定義、事件追蹤設計、KPI 表。

### L7–L8：AI Orchestrator PM／AI 產品架構 PM

- **目標**：  
  - 將需求放進整體 AI 能力平台，決定該做成模組還是共用服務。  
- **建議作法**：  
  - 用能力地圖（Capability Map）討論這個需求位於哪一格；  
  - 決定未來是否可重用到其他產品線。  
- **產出物**：  
  - 能力地圖、架構圖、Roadmap 節點。

### L9–L10：組織級 AI Enablement／AI-First 領航者

- **目標**：  
  - 對齊策略、治理與投資優先順序。  
- **建議作法**：  
  - 一頁紙說明：這條 AI 工作流對營收、成本、風險的影響；  
  - 提出 2–3 個可量化的「投資換回報」場景。  
- **產出物**：  
  - 商業案例、投資組合視圖、治理原則摘要。

## 4. 在 R0 工作流中的角色

在整個 AI Orchestrator 工作流中，本篇扮演的角色是：

- 幫助 **R0 / AI PM** 在專案一開始就：
  - 評估利害關係人目前的 AI 使用成熟度；  
  - 選擇適當深度與溝通載體；  
  - 設定每一場會議「現實可達成」的目標。
